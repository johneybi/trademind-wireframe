## 1. 제품 개요

### 1.1 한 줄 정의

TRADEMIND는 AI 기반 질문을 통해 투자자의 감정적 판단을 늦추고, 스스로 이유를 다시 보게 돕는 투자 멘탈 케어 서비스입니다.

### 1.2 이 서비스가 하는 일 / 하지 않는 일

하는 일
- 투자 직전과 직후의 감정적 흔들림에 개입한다.
- CBT 기반 질문으로 감정과 근거를 분리해 보게 만든다.
- 비슷한 상황의 집단 맥락을 통해 심리적 고립감을 낮춘다.
- 사전 점검과 사후 복기를 기록으로 남겨 반복 패턴을 보게 한다.

하지 않는 일
- 종목 추천
- 매수/매도 지시
- 수익률 예측
- 자산 연동 기반의 투자 관리 앱 역할

### 1.3 해커톤 주제 적합성

| 축 | 구현 방식 |
| --- | --- |
| AI | 인지 왜곡 유형을 분류하고, 맞춤형 객관화 질문을 생성 |
| TOGETHER | 익명 집단 경험 데이터를 통해 비슷한 상황의 심리 맥락 제공 |
| Human Well-Being | 충동적 투자 판단을 늦추고, 감정 건강과 판단 건강을 함께 다룸 |

## 2. 핵심 사용자 문제

- 사용자는 투자 직전과 직후에 감정과 판단을 구분하지 못한 채 행동한다.
- 시장 급등락 상황에서 FOMO, 불안, 매몰비용, 확증편향 같은 인지 왜곡이 반복된다.
- 투자 앱은 많지만, 감정적 판단을 늦추는 서비스는 거의 없다.

핵심 질문:
- 지금 내 판단이 근거 때문인지 감정 때문인지 알 수 있을까?
- 이미 한 선택을 다음번에는 다르게 보기 위해 기록으로 남길 수 있을까?

## 3. 정보 구조

### 3.1 하단 탭 구조

MVP는 3탭 구조로 간다.

| 탭 | 역할 |
| --- | --- |
| 홈 | 서비스 진입, 오늘의 시장 감정, 감정 공유 보드 미리보기 |
| 대화하기 | 입력, AI 대화, 결과까지 이어지는 핵심 플로우 |
| 내 기록 | 멘탈 캘린더 중심의 개인 기록 화면 |

설정 탭은 MVP에서 제외한다.  
대신 홈 우상단 계정 진입점으로 로그인/회원가입을 제공한다.

### 3.2 화면 목록

| 화면 | 역할 |
| --- | --- |
| 홈 | 서비스 소개, 오늘의 감정 흐름, 감정 공유 보드 미리보기 |
| 입력 0depth | 대화 시작 전 진입 화면 |
| 입력 Step 1~4 | 단계형 입력 플로우 |
| AI 채팅 | CBT 기반 질문 플로우 |
| 결과 | 사전/사후 결과 variant를 가진 단일 결과 셸 |
| 내 기록 | 멘탈 캘린더 단일 화면 |
| 계정 진입 / 로그인 | 홈 우상단에서 접근하는 보조 흐름 |

## 4. 핵심 사용자 흐름

### 4.1 사전 모드

1. 사용자가 `사고 싶거나 팔고 싶어요`를 선택한다.
2. 종목, 감정, 상황을 단계별로 입력한다.
3. 브릿지 화면에서 AI 대화 시작 전 흐름을 정리한다.
4. AI가 인지 왜곡을 바탕으로 질문을 던진다.
5. 사용자는 마지막에 `잠시 관망하겠습니다` 또는 `그래도 진행하겠습니다`를 선택한다.
6. 선택은 현재 화면 안에서 확인 상태로 닫힌다.

### 4.2 사후 모드

1. 사용자가 `이미 한 매매가 마음에 걸려요`를 선택한다.
2. 종목, 감정, 상황을 단계별로 입력한다.
3. 브릿지 화면에서 복기 대화 진입 전 흐름을 정리한다.
4. AI가 복기 질문을 던진다.
5. 결과 화면 진입 시 복기 기록은 이미 저장된 상태로 간주한다.
6. 사용자는 `멘탈 캘린더 보러 가기`를 통해 내 기록으로 이동할 수 있다.

### 4.3 브릿지 단계의 역할

브릿지 단계는 입력을 끝내고 AI 대화로 넘어가기 직전의 전환 화면이다.

역할:
- 입력 단계와 채팅 단계를 분리
- 방금 적은 내용을 정서적으로 정리
- 실제 제출 시점을 명확히 만듦

## 5. 화면 요구사항

### 5.1 입력 화면

- 0depth와 1depth 이후는 같은 셸이 아니다.
- 0depth는 탭바가 보이는 진입 화면이다.
- 1depth 이후는 스텝 헤더와 CTA를 가진 단계형 플로우다.
- 한 번에 하나의 질문만 보여준다.
- 입력 요소는 설명보다 먼저 보이게 배치한다.

### 5.2 AI 채팅 화면

- 사전/사후는 서로 다른 시나리오지만 같은 채팅 셸을 공유한다.
- 첫 사용자 말풍선은 입력 화면에서 전달된 맥락이라는 단서가 보여야 한다.
- AI 응답 중에는 입력창이 잠긴다.
- 프론트는 `mode + stage`로 흐름을 제어한다.
- 질문의 성격은 `question_type` enum으로 구분한다.

### 5.3 결과 화면

- 결과 화면은 사전/사후 variant를 가진 단일 셸이다.
- 사전 결과는 `최종 선택 확정`이 핵심이다.
- 사후 결과는 `이미 저장된 복기 기록의 다음 이동`이 핵심이다.
- 사전의 두 선택은 서로 다른 판단이지만 동일한 방식으로 닫힌다.
- 결과 CTA 이후 별도 새 화면으로 넘기지 않고, 현재 화면 안에서 확인 상태로 닫는다.

사전 결과
- CTA: `잠시 관망하겠습니다` / `그래도 진행하겠습니다`
- 의미: 분석 결과를 본 뒤, 사용자가 스스로 최종 방향을 언어화한다.

사후 결과
- CTA: `멘탈 캘린더 보러 가기`
- 의미: 저장 버튼이 아니라, 이미 저장된 복기 기록에서 내 기록 화면으로 이동하는 버튼이다.

### 5.4 홈 화면

- 오늘의 시장 감정 요약을 보여준다.
- 감정 공유 보드는 홈 화면의 일부로 노출한다.
- 익명 사연과 집단 맥락은 홈에서 미리보기 형태로 보여준다.

### 5.5 내 기록 화면

목적:
- 사전 점검과 사후 복기 기록이 언제 남았는지 보고, 그날의 상세 내용을 다시 보는 개인 기록 화면

중요 원칙:
- 감정 공유 보드는 내 기록 안에 넣지 않는다.
- 내 기록은 멘탈 캘린더 중심의 개인 화면이다.
- 읽기형 화면이므로 헤더를 고정하지 않는다.

멘탈 캘린더가 보여줘야 하는 것:
- 기록이 남은 날짜의 흐름
- 어느 시기에 점검이나 복기가 몰렸는지
- 필요하면 같은 날 여러 기록이 있었는지를 진하기 차이로만 표현하는 것

멘탈 캘린더가 메인으로 보여주지 않아야 하는 것:
- 감정이나 왜곡의 상세 내용을 셀 하나에 직접 설명하는 것
- `관망 / 진행 / 기록`을 같은 수준의 분류로 나열하는 것

이유:
- 히트맵은 상세 내용을 읽는 장치가 아니라, 기록이 남은 시기를 한눈에 보는 장치다
- `관망 / 진행`은 사전 모드의 최종 선택
- `기록`은 사후 모드의 저장 완료 상태
- 셋은 같은 축의 값이 아니며, 그대로 노출하면 사용자에게 해석 가치가 낮다

즉:
- 히트맵의 메인 값은 `그날 기록이 있었는지`다
- 같은 날 기록이 여러 개면 강도를 더 진하게 줄 수는 있지만, 의미는 여전히 `기록의 밀도`에 머문다
- 감정, 왜곡, 요약은 아래 리스트나 상세 보기에서 읽는다
- `관망 / 진행 / 기록`은 캘린더의 메인 표시값이 아니며, 필요하면 상세 보기에서만 보조 메타데이터로 노출한다

히트맵 아래 보조 섹션:
- `자주 걸리는 생각`
  - 비슷한 순간마다 반복되는 생각 패턴을 짧게 보여주는 영역
- `마음이 남았던 순간들`
  - 최근 기록 중 어떤 마음이 남았는지 사례로 다시 보는 영역

## 6. AI 계약

### 6.1 입력 데이터

```json
{
  "mode": "pre" | "post",
  "ticker": "삼성전자" | null,
  "emotion": "불안해요",
  "text": "3일 연속 빠지는데 더 떨어질 것 같아요..."
}
```

### 6.2 LLM 출력 계약

```json
{
  "distortion_type": "catastrophizing",
  "distortion_tag": "최악을 상상하고 있어요",
  "empathy": "3일 연속 하락이면 정말 불안하셨겠어요.",
  "question": "처음 이 종목을 봤을 때의 근거가 오늘도 유효한가요?",
  "question_type": "exploratory",
  "meta_question": "지금 이 판단이 근거 때문인가요, 감정 때문인가요?",
  "meta_question_type": "meta_cognition",
  "meta_options": ["근거 때문이에요", "감정 때문인 것 같아요"]
}
```

역할 분리:
- LLM: 질문 내용과 `question_type` 반환
- 백엔드: `question_type`을 받아 `required` 계산
- 프론트: `mode + stage`로 흐름 제어

### 6.3 question_type enum

| 값 | 의미 |
| --- | --- |
| `exploratory` | 건너뛰기 가능한 탐색 질문 |
| `meta_cognition` | 최종 선택 전 메타인지 질문 |

## 7. 데이터 및 기록 원칙

### 7.1 결과 저장 의미

사전 결과
- 분석 결과는 결과 화면 진입 시점에 준비된 상태로 본다
- 최종 선택은 `PAUSE` 또는 `PROCEED`로 별도 확정된다

사후 결과
- 결과 화면 진입 시점에 복기 기록이 저장된 상태로 본다
- `멘탈 캘린더 보러 가기`는 저장이 아니라 이동이다

### 7.2 내부 데이터 모델 해석 원칙

`관망 / 진행 / 기록`을 하나의 동일 축 상태값으로 해석하지 않는다.

더 적절한 해석:
- `session_mode = pre | post`
- `pre_decision = pause | proceed | null`
- `post_review_completed = true | false`

즉,
- `관망 / 진행`은 사전 선택 데이터
- `사후 기록 완료`는 별개의 완료 상태

멘탈 캘린더 해석 원칙:
- 히트맵은 `그날 기록이 남았는지`를 보여주는 표시다
- 같은 날 여러 기록이 있으면 강도를 더 진하게 줄 수 있다
- 감정, 왜곡, 요약은 상세 리스트로 내려서 읽는다

### 7.3 대시보드/집계 활용 가능성

의미 있는 활용:
- 사전 모드 진입 대비 관망 선택 비율
- 인지 왜곡 유형별 최종 선택 분포
- 사전에서 진행 후 나중에 사후 복기로 돌아오는 비율
- 감정/왜곡의 반복 시점

의미 없는 활용:
- `관망 / 진행 / 기록`을 동일 레벨로 묶어 단일 차트로 비교하는 것

## 11. v1.2 변경 사항

- 멘탈 캘린더의 의미를 재정의했다.
- `관망 / 진행 / 기록`을 동일한 사용자 표시값으로 쓰지 않기로 정리했다.
- `관망 / 진행`과 `사후 기록 완료`를 분리된 데이터 의미로 다루도록 명시했다.
- 내 기록은 멘탈 캘린더 단일 화면이며, 감정 공유 보드는 홈에 둔다고 다시 고정했다.
- 결과 화면의 저장/이동 의미를 다시 명확히 정리했다.
