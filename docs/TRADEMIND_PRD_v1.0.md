## 1. 제품 개요

### 1.1 한 줄 정의

> TRADEMIND는 AI 기반 질문을 통해 투자자의 감정적 판단을 늦추고, 스스로 이유를 다시 보게 돕는 투자 멘탈 케어 서비스입니다.

### 1.2 제품 진실 — 하는 것 / 안 하는 것

**하는 것**
- 매매 충동이 오는 순간(사전) 또는 충동적 매매 직후(사후), CBT 기반 AI 대화로 개입
- 인지 왜곡 패턴을 탐지하고 소크라테스식 질문으로 사용자 스스로 판단을 점검하게 함
- 익명 집단 데이터("비슷한 상황의 투자자 78%가 관망을 선택")로 심리적 안전감 제공
- 사후 기록 축적 → 패턴 인식 → 선제 개입의 학습형 순환 구조 제공

**안 하는 것**
- 종목 추천 또는 매수/매도 지시
- 수익률 예측 또는 금융 조언
- 증권사 계좌/자산 연동
- 사용자 투자 성과 평가 또는 순위 비교

### 1.3 해커톤 주제 적합성

| 키워드 | 구현 방식 |
| --- | --- |
| **AI** | CBT(인지행동치료) 프레임 기반 LLM이 투자자의 인지 왜곡 패턴을 실시간 분석하고, 맞춤형 객관화 질문을 제공 |
| **TOGETHER** | 익명 감정 공유 보드를 통해 "비슷한 상황의 투자자들은 어떤 선택을 했는가"를 보여줌으로써, 개인의 고립된 판단을 집단의 경험으로 보완. "혼자가 아니다"는 심리적 안전감 제공 |
| **Human Well-Being** | 투자는 이성의 영역처럼 보이지만, 실제 손실의 대부분은 잘못된 정보가 아닌 **감정적 판단**에서 발생한다. 개인 투자자의 매매 실패 원인 1위는 정보 부족이 아닌 **충동적 의사결정**이며, TRADEMIND는 그 충동의 근원인 인지 왜곡을 CBT 프레임으로 교정함으로써 투자자의 재정 건강과 심리 건강을 동시에 지킨다. |

## 2. 사용자

### 2.1 타겟

20~40대 개인 투자자. 경제 기사나 시황을 챙겨보며 나름의 원칙이 있으나, 실전 호가창 앞에서는 감정 통제가 안 되어 잦은 매매와 손실을 반복하는 **'지식형 뇌동매매자'**.

### 2.2 페르소나

- **프로필:** 33세, 직장인(대리), 하루 평균 주식 앱 접속 15회 이상
- **상황:** 우량주 장기 투자를 결심했으나, 며칠 연속 파란불이 켜지면 업무에 집중하지 못하고 화장실에서 매도를 고민함. 밤에 미국 장에서 특정 밈 주식이 급등하면 "나만 벼락거지 될 것 같다"는 조급함에 원칙을 깨고 추격 매수 버튼에 손을 올림
- **핵심 니즈:** "내가 지금 이성적인 건지, 감정에 휘둘리는 건지 누군가 객관적으로 짚어줬으면"

## 3. 핵심 방법론

### 3.1 심리치료 기법 4가지 기반

| 기법 | 원래 쓰임 | TRADEMIND 적용 |
| --- | --- | --- |
| **타당화 (Validation)** | DBT의 핵심. 감정을 교정하기 전, 그 감정이 당연하다는 것을 100% 인정 | AI가 항상 공감 메시지로 시작. "피 같은 돈이 줄어들어 불안한 건 당연해요"처럼 감정 자체를 수용 |
| **인지 재구성 — ABC 모델** | 사건(A) → 비합리적 신념(B) → 감정/행동(C) 연결 고리를 끊고, 논박(D)으로 합리적 신념(E)으로 교정 | LLM이 사용자 입력을 A(객관적 사실)와 B(왜곡된 해석)로 분리 후, 소크라테스식 질문(D)으로 모순을 발견하게 유도 |
| **소크라테스식 문답법** | "틀렸어"가 아닌 질문으로 상대가 스스로 논리의 허점을 발견하게 만드는 기법 | AI는 직접 "팔지 마세요"를 하지 않음. "15% 오른 종목이 다음날도 오른 경우가 더 많을까요?"처럼 스스로 답을 찾도록 유도 |
| **그라운딩 (Grounding)** | 극도의 불안 상태에서 현재로 돌아오게 하는 즉각적 안정화 기법 | 감정 기록(타이핑 브레이크) 자체가 그라운딩. 공포를 텍스트로 변환하는 행위가 뇌의 흥분 상태를 진정시킴 |

### 3.2 투자 맥락 인지 왜곡 8유형

| # | 유형 | 정의 | 투자 맥락 예시 | 전형적 결과 |
| --- | --- | --- | --- | --- |
| 1 | **파국화 (Catastrophizing)** | 최악의 시나리오만 상상하며 현실을 과대평가 | "이 주식 더 떨어지면 전 재산 날린다" | 패닉셀 |
| 2 | **흑백 사고 (All-or-Nothing)** | 중간 지대 없이 극단적으로만 판단 | "지금 안 사면 영원히 기회 없다" | FOMO 추격매수 |
| 3 | **감정적 추론 (Emotional Reasoning)** | 감정을 객관적 근거로 착각 | "불안하니까 분명 뭔가 잘못된 거야" | 근거 없는 매도 |
| 4 | **확증 편향 (Confirmation Bias)** | 자신의 결론을 지지하는 정보만 수집 | "이 종목 오를 수밖에 없어" (반대 근거 무시) | 과도한 집중투자 |
| 5 | **소외 불안 + 군집 행동 (FOMO + Herd Behavior)** | 타인의 수익에서 나만 배제된다는 사회적 박탈감 | "남들은 다 버는데 나만 못 타면 어쩌죠" | 고점 추격매수 |
| 6 | **통제의 환상 (Illusion of Control)** | 무작위적 시장을 자신이 통제할 수 있다는 자기기만 | "내가 계속 보고 있어서 괜찮다" | 과잉매매, 손절 실패 |
| 7 | **기준점 편향 (Anchoring Bias)** | 처음 인식한 가격에 묶여 현재 가치를 무시 | "최고점 10만 원이었으니 지금 6만 원은 무조건 싸다" | 펀더멘털 무시한 물타기 |
| 8 | **매몰비용 오류 (Sunk Cost Fallacy)** | 회수 불가능한 손실에 집착해 합리적 결정을 내리지 못함 | "여기까지 물렸는데 억울해서 어떻게 손절하냐" | 손절 실패, 시드머니 묶임 |

### 3.3 변화 단계 모형 기반 AI 개입 전략 (TTM)

사용자가 어느 심리 단계에 있는지에 따라 AI 톤이 달라진다.

| 단계 | 사용자 심리 | AI 전략 |
| --- | --- | --- |
| **계획 이전** (부인) | "내 매매 방식엔 문제없어" | 공감 우선, 비판 금지. 개방형 질문으로 미세한 의구심만 심어줌 |
| **계획** (양가감정) | "뭔가 잘못된 것 같은데, 그래도 지금 사야 할 것 같아" — **앱을 켠 대부분의 사용자가 이 단계** | CBT 개입 효과가 가장 큰 황금시간대. ABC 모델 + 소크라테스식 질문으로 인지 왜곡 교정 집중 |
| **준비** (결심) | 충동이 가라앉고 쿨다운을 진지하게 고민하는 순간 | 커뮤니티 데이터 제시로 결단 촉진 |
| **행동 + 유지** | 관망을 선택하고 건강한 투자 습관을 형성해 나가는 단계 | 긍정적 강화. 멘탈 캘린더로 성공 시각화. 재발도 학습으로 프레이밍 |

### 3.4 TOGETHER의 임상적 가치 — 역-군집 효과

- **보편성(Universality) 효과:** "나만 이런 충동을 느끼는 게 아니다"라는 자각만으로 고립감과 자책감이 해소됨
- **역-군집(Reverse-herd) 효과:** FOMO는 "다들 사고 있다"는 환상을 먹고 자람. 공유 보드는 "이성을 찾고 쿨다운을 선택한 건강한 다수"를 실시간으로 보여줌으로써 군집 심리의 에너지를 역이용
- **심리적 안전감:** "나와 같은 상황에서 78%가 관망을 선택했다"는 데이터가 혼자 호가창 앞에 선 투자자 옆에 앉아주는 역할

## 5. 기능 명세

### 5.1 홈 화면

**목적:** 서비스의 첫 인상. 핵심 CTA로 빠르게 진입하게 하고, TOGETHER 분위기를 미리 전달함.

**구성 요소:**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 로고 + 슬로건 | 예: "지금 그 충동, 잠깐 멈춰요" | — |
| 계정 진입점 | 우상단 아이콘 👤 | 로그인/내 기록 연결 |
| 시장 감정 한 줄 | 예: "지금 투자자 68%가 FOMO 상태입니다" | 홈 화면 감정 공유 보드의 집계 데이터를 요약하여 표시 |
| 핵심 CTA | [살까?] / [팔까?] 버튼 | 대화하기 탭과 동일한 진입점 |
| 익명 한줄 미리보기 | 1~2개 | TOGETHER 분위기 전달 |
| 하단 탭바 | [홈] [대화하기] [내 기록] | 3탭 고정 |

### 5.2 입력 화면 (스텝 플로우)

**목적:** 폼 작성이 아닌 감정 흐름 설계. 멈추고 → 감정에 이름 붙이고 → 이유를 적고 → 대화로 넘어가는 경험.

> **v0.6 → v0.7 변경 이유:** v0.6의 단일 화면은 데이터 수집 관점에서는 효율적이었으나, 감정이 큰 순간에 여러 입력을 한 번에 보이면 인지 부하가 크고 서비스 톤과 맞지 않았음. v0.7에서는 사용자의 감정 흐름을 설계하는 관점으로 전환, 스텝 플로우로 재구성.

> **구조 원칙:** 0depth 진입 화면과 Step 1~4 입력 플로우는 UX 셸이 다르므로, 개발 구조상 별도 페이지 단위로 분리한다. 다만 프로토타입 인덱스에서는 하나의 흐름으로 이어서 프리뷰할 수 있다.

**[0depth] 진입 화면**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 분기 버튼 | [사고 싶거나 팔고 싶어요] / [이미 한 매매가 마음에 걸려요] | 사전/사후 모드 분기 기준. 탭바 노출 |

**[Step 1] 종목 단계**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 종목명 입력 | 자동완성 + Fallback 자유 텍스트 | 백엔드 stocks 배열 기반 자동완성. 목록 외 종목은 자유 텍스트 직접 입력 가능 |
| 예외 처리 | 종목 인식 불가 시: "해당 종목 데이터를 가져올 수 없지만, 지금 느끼시는 감정에 집중해서 대화를 이어가 볼까요?" | — |

**[Step 2] 감정 단계**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 감정 버튼 | [불안해요] [조급해요] [확신해요] [후회돼요] [모르겠어요] | 단일 선택 |

**[Step 3] 상황 입력 단계**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 자유 텍스트 | "조금 더 자세히 적어주세요." | 선택 입력 |

**[Step 4] 브릿지 단계**

> **정의:** 사용자가 적은 내용을 바탕으로 AI 대화가 시작되기 직전, 흐름을 정서적으로 정리하고 제출 시점을 명확히 만드는 전환 화면.

| 역할 | 설명 |
| --- | --- |
| 심리적 완충 구간 | "이제 하나씩 살펴볼게요"처럼 입력과 채팅 사이의 자연스러운 전환 |
| commit point | 브릿지 CTA를 누르는 순간 세션 생성 및 데이터 전송. 프론트에서 로컬 상태 → 서버로 commit |
| 탭바 미노출 | step 화면 전체에서 탭바 미노출, 상단 스텝 인디케이터 + 하단 CTA 구조 |

### 5.3 AI 채팅 화면

**목적:** CBT 기반 AI 대화로 인지 왜곡을 스스로 발견하게 함. 하나의 채팅 셸 안에서 사전/사후 시나리오 분기.

**채팅 계약 요약:**

| 요소 | 설명 |
| --- | --- |
| 상단 태그 | 첫 AI 응답 후 표시. 내부 코드는 인지 왜곡 유형, 사용자에게는 일상 언어로 표시 (예: ⚡ 지금 아니면 안 된다는 생각) |
| LLM 책임 | `distortion_type`, `distortion_tag`, `empathy`, `question`, `question_type`, `meta_question`, `meta_question_type`, `meta_options` 반환 |
| 백엔드 책임 | `question_type`, `meta_question_type`을 받아 `required` 계산 및 응답 구조 정리 |
| 프론트 책임 | `mode + stage`로 화면 흐름 제어, `required`로 건너뛰기 버튼 노출 여부 제어 |
| MVP 원칙 | 자유응답을 길게 반복하지 않고, 질문 수를 줄여 응답 품질 편차와 상태 복잡도를 낮춘다 |

**사전 모드 (현실적 MVP 구조):**

| 단계 | 내용 | 사용자 액션 |
| --- | --- | --- |
| 1 | AI가 `공감 1문장 + 객관화 질문 1문장`을 제시 | 자유 텍스트 응답 또는 건너뛰기 |
| 2 | AI가 `메타 질문 + 2개 선택지`를 제시 | 버튼 1개 필수 선택 |
| 3 | 결과 화면으로 이동 | — |

**사후 복기 모드 (현실적 MVP 구조):**

| 단계 | 내용 | 사용자 액션 |
| --- | --- | --- |
| 1 | AI가 `공감 1문장 + 복기 질문 1문장`을 제시 | 자유 텍스트 응답 또는 건너뛰기 |
| 2 | 결과 화면으로 이동 | — |

**AI 응답 원칙:**
- 타당화 먼저: 감정을 교정하기 전 반드시 공감 1문장으로 시작
- 판단하지 않는다: "그건 잘못된 판단이에요" 금지. "많이 불안하시겠어요" (O)
- 전문 용어 금지: "파국화", "인지 왜곡" 등은 말풍선에 노출하지 않음. 상단 태그에만 일상 언어로 표시
- 가드레일: 종목 추천, 매수/매도 지시, 수익률 예측은 시스템 프롬프트에서 명시적으로 차단

---

### 5.4 결과 화면

**목적:** 사용자의 현재 판단을 정리하고, 집단 맥락을 제공하고, 선택을 하나의 행동으로 확정하게 함. 하나의 셸 안에서 사전/사후 variant 분기.

**핵심 원칙:**
- CTA는 이동 버튼이 아니라 **선택 확정 버튼**
- 관망과 진행은 서로 다른 판단이지만 **동일한 방식으로 닫힌다**
- 버튼 선택 직후 별도 화면으로 이동하지 않고 **현재 화면 안에서 확인 상태로 전환**
- `그래도 진행하겠습니다`는 막지 않고, 응원하지도 않으며, 기록만 남긴다
- 사후 모드에서는 결과 화면이 뜰 때 이미 복기 기록이 저장된 상태를 전제로 본다

**사전 모드 (pre variant):**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 헤드카피 | "한 발 물러서 보니 어떠세요?" | — |
| 상단 태그 유지 | 인지 왜곡 일상 언어 태그 | — |
| 커뮤니티 데이터 | "비슷한 상황의 투자자 78%가 관망을 선택" | 같은 인지 왜곡 유형 필터링 |
| 익명 한줄 이유 | 2~3개 | 더미 데이터 (MVP) |
| 상태 안내 | "지금 보고 있는 요약은 이미 정리됐어요. 마지막 선택만 남았어요." | 분석 결과는 결과 화면 진입 시점에 준비 완료 |
| CTA | [잠시 관망하겠습니다] / [그래도 진행하겠습니다] | 저장값: PAUSE / PROCEED |
| 확인 상태 전환 | 버튼 선택 직후 현재 화면 안에서 전환 | 별도 화면 없음 |

**사후 모드 (post variant):**

| 요소 | 설명 | 비고 |
| --- | --- | --- |
| 헤드카피 | "오늘의 기록이 다음번을 바꿔줄 거예요." | — |
| 상단 태그 유지 | 인지 왜곡 일상 언어 태그 | — |
| 커뮤니티 데이터 | "비슷한 상황에서 사후 후회를 느낀 투자자 비율" | — |
| 상태 안내 | "이번 복기는 이미 저장됐어요." | CTA는 저장이 아니라 다음 화면 이동 |
| CTA | [멘탈 캘린더 보러 가기] | 저장값: POST_RECORDED |
| 확인 상태 전환 | 버튼 선택 직후 현재 화면 안에서 전환 | 별도 화면 없음 |

**확인 상태 처리 원칙:**
- 사전 모드는 `최종 선택 확정`이므로 버튼 클릭 후 현재 화면 안에서 확인 상태로 닫는다
- 사후 모드는 `이미 저장된 기록의 다음 이동`이므로, CTA는 저장 버튼이 아니라 멘탈 캘린더 이동 버튼으로 해석한다
- MVP 프로토타입에서는 실제 라우팅 대신 현재 화면 안에서 짧은 전환 상태로 표현할 수 있다

---

### 5.5 내 기록 화면

**목적:** 사용자의 사전/사후 기록을 한곳에서 다시 보고, 반복되는 감정과 인지 왜곡 패턴을 확인하는 개인 기록 화면. 읽기형 화면이므로 헤더 고정하지 않음.

> 감정 공유 보드는 내 기록 안에 넣지 않고, 홈 화면에서 오늘의 시장 감정과 익명 사연 미리보기로 제공한다.

**멘탈 캘린더:**

| 요소 | 설명 |
| --- | --- |
| 월별 히트맵 | PAUSE / PROCEED / POST_RECORDED 날짜별 표시 |
| 인지 왜곡 유형별 빈도 차트 | 바 차트 — 내가 가장 자주 반복하는 패턴 시각화 |
| 최근 기록 리스트 | 날짜 / 유형 태그 / 선택 결과 |
| 기록 누적 구조 | 사전 점검과 사후 복기 기록이 같은 캘린더 안에서 누적 |

## 6. AI 설계

### 6.1 입력 데이터 구조

화면 2에서 사용자가 입력한 값이 시스템 프롬프트에 전달된다.

```json
{
  "mode": "pre" | "post",
  "ticker": "삼성전자" | null,
  "emotion": "불안해요",
  "text": "3일 연속 빠지는데 더 떨어질 것 같아요..."
}
```

- `mode`: 화면 2 분기 버튼 선택값. LLM 자동 판별 불필요
- `ticker`: 종목명 (선택 입력, null 허용)
- `emotion`: 감정 버튼 선택값 (단일 선택)
- `text`: 자유 텍스트 (선택 입력)

### 6.2 인지 왜곡 탐지 기준

| # | 내부 코드 | 사용자 태그 | 탐지 핵심 키워드/패턴 |
| --- | --- | --- | --- |
| 1 | `catastrophizing` | 🌀 최악을 상상하고 있어요 | "전 재산 날린다", "반토막", "다 잃을 것 같다" |
| 2 | `all_or_nothing` | ⚡ 지금 아니면 안 된다는 생각 | "지금 안 사면", "영원히 기회 없다", "이번이 마지막" |
| 3 | `emotional_reasoning` | 💭 감정이 판단을 이끌고 있어요 | "불안하니까", "느낌이 안 좋아서", "왠지 떨어질 것 같아" |
| 4 | `confirmation_bias` | 🔍 보고 싶은 것만 보이고 있어요 | "오를 수밖에 없어", "악재는 찌라시야" |
| 5 | `fomo_herd` | 👥 나만 뒤처질 것 같은 느낌 | "남들은 다 버는데", "나만 못 타면", "다들 산다는데" |
| 6 | `illusion_of_control` | 🎯 내가 컨트롤할 수 있다는 믿음 | "내가 계속 보고 있어서", "이번엔 내 느낌이 맞다" |
| 7 | `anchoring_bias` | 📌 처음 본 가격에 묶여 있어요 | "최고점이 OO원이었는데", "원래 OO원이었으니까 싼 거다" |
| 8 | `sunk_cost` | 🪤 지금까지 한 게 아까운 마음 | "여기까지 물렸는데", "억울해서 못 팔겠다", "본전은 찾아야지" |

### 6.3 JSON 출력 스키마

LLM은 아래 JSON 형태로만 응답한다.

```json
{
  "distortion_type": "catastrophizing",
  "distortion_tag": "🌀 최악을 상상하고 있어요",
  "empathy": "3일 연속 하락이면 정말 불안하셨겠어요.",
  "question": "처음 이 종목을 매수했을 때의 근거가 오늘도 여전히 유효한가요?",
  "question_type": "exploratory",
  "meta_question": "지금 매도하려는 이유가 '매수 근거가 바뀌어서'인가요, 아니면 '빨간 숫자가 너무 불안해서'인가요?",
  "meta_question_type": "meta_cognition",
  "meta_options": ["매수 근거가 바뀌었어요", "감정 때문인 것 같아요"]
}
```

**역할 분리 원칙:**
- `LLM`
  - 질문 내용과 질문 성격을 `question_type`, `meta_question_type`으로 반환
- `백엔드`
  - enum을 받아 `required`를 계산
  - `exploratory` → 건너뛰기 가능
  - `meta_cognition` → 건너뛰기 불가
- `프론트`
  - `mode + stage`로 채팅 흐름을 제어
  - `required`로 버튼 노출 여부를 제어

즉, MVP에서는 `흐름 제어`와 `질문 성격 제어`를 분리한다.

### 6.4 LLM 일관성 확보 방안

- **시스템 프롬프트:** 8가지 인지 왜곡 유형 정의, 탐지 기준, 대응 질문 템플릿 포함
- **Few-shot 예시:** 각 유형별 1~2개 입출력 예시 삽입하여 응답 톤과 구조 일관성 확보
- **가드레일:** 종목 추천, 매수/매도 지시, 수익률 예측은 시스템 프롬프트에서 명시적으로 차단
- **사전/사후 분기:** 화면 2 분기 버튼 선택값을 시스템 프롬프트에 명시적으로 전달. LLM 자동 판별 불필요
- **MVP 계약 최소화:** 해커톤 단계에서는 LLM 출력 계약을 `question_type` 수준까지만 유지하고, 복잡한 질문 상태 분기는 프론트에서 `mode + stage` 기준으로 제어

### 6.5 AI 응답 톤 가이드

- **타당화 먼저:** 공감 1문장 → 질문 1문장 원칙. 감정 교정 전 반드시 수용
- **판단하지 않는다:** "많이 불안하시겠어요" (O) / "그건 잘못된 판단이에요" (X)
- **전문 용어 금지:** 말풍선에서는 일상 언어만. 상단 태그에만 표시
- **차분하고 건조한 톤:** 과도한 위로나 감정적 표현 지양
- **유도 질문 금지:** 정답이 정해진 질문 금지. 사용자 스스로 발견하게 설계

## 7. 데이터 설계

### 7.1 저장값 Enum

UI 문구와 내부 저장값은 분리한다. 카피 수정이 자유롭고 집계 및 대시보드 설계가 안정적이어야 하기 때문.

| Enum | 의미 | 발생 화면 |
| --- | --- | --- |
| `PAUSE` | 관망 선택 (사전 모드) | 결과 화면 (pre) |
| `PROCEED` | 진행 선택 (사전 모드) | 결과 화면 (pre) |
| `POST_RECORDED` | 사후 복기 기록 저장 | 결과 화면 (post) |

### 7.2 세션 데이터 구조 (예시)

```json
{
  "session_id": "uuid",
  "uid": "firebase_uid",
  "created_at": "2026-04-02T11:30:00Z",
  "mode": "pre",
  "ticker": "삼성전자",
  "emotion": "불안해요",
  "text": "3일 연속 빠지는데...",
  "distortion_type": "catastrophizing",
  "result": "PAUSE"
}
```

### 7.3 대시보드 활용 가능 지표

**개인 지표 (멘탈 캘린더):**
- 쿨다운 전환율: 사전 모드 진입 대비 PAUSE 비율
- 진행 선택 비율: 어떤 상황에서 계속 밀고 가는지
- 인지 왜곡 유형별 선택 패턴
- 사전 → 사후 회귀 흐름: PROCEED 후 POST_RECORDED로 돌아오는 비율
- 시간대/요일 패턴

**집단 지표 (감정 공유 보드):**
- 같은 왜곡 유형에서 PAUSE / PROCEED 비율
- 특정 감정 상태의 빈도
- 사후 후회 비율

**대시보드 원칙:** 사용자의 투자 성과를 평가하는 도구가 아니라, 반복되는 감정과 행동 패턴을 비추는 거울이어야 한다. 수익률 기반 우월감/열등감, 사용자 간 순위 비교는 표시하지 않는다.

## 8. 기술 스택

| 구분 | 내용 | 비고 |
| --- | --- | --- |
| 타입 | 모바일 웹 (반응형) | — |
| 프론트엔드 | React + Tailwind CSS | — |
| 백엔드 | Spring (Java) | SSE(Server-Sent Events)로 AI 스트리밍 응답 처리 |

| 인증 | Firebase Authentication (이메일/비밀번호) | JWT 없이 세션(쿠키) 방식. 최대한 컴팩트하게 구현 |
| LLM | 미정 | OpenAI / Gemini 등 후보 검토 중 |
| AI 통신 | SSE (Server-Sent Events) | 웹소켓 불필요. 단방향 스트리밍으로 충분 |

**기술적 차별화 포인트:**
- **CBT 구조화 프롬프트:** 인지 왜곡 유형 분류 → 맞춤 질문 생성의 2단계 파이프라인
- **JSON 스키마 응답 강제:** `{distortion_type, distortion_tag, empathy, question, question_type, meta_question, meta_question_type, meta_options}` 구조로 강제. `question_type` enum을 백엔드가 `required`로 매핑하여 프론트 버튼 제어
- **익명 집계 파이프라인:** 사용자 선택값 + 인지 왜곡 유형을 집계하여, 개인 식별이 불가능한 통계만 공유 보드에 노출

## 9. 비기능 요구사항

### 9.1 보안 및 프라이버시

- 증권사 계좌, 보유 종목, 자산 정보 등 금융 데이터는 **일절 수집하지 않음**

## 10. 유저 시나리오

이 섹션은 발표 대본이 아니라, 사용자가 어떤 맥락에서 TRADEMIND를 열고 어떤 도움을 받는지를 설명하는 제품 시나리오입니다.

### 시나리오 A. 급등장에서 따라 들어가고 싶을 때

**상황**
- 사용자는 밤 늦게 급등 중인 해외 종목을 보고 조급해진다.
- 머리로는 추격매수가 위험하다는 걸 알지만, `지금 안 사면 놓칠 것 같다`는 감정이 커진다.

**사용자 목표**
- 지금 들어가도 되는지 누가 대신 판단해주길 원하는 것이 아니라,
- 이 감정이 실제 근거에서 나온 것인지 잠깐 점검하고 싶다.

**서비스 흐름**
- 사용자는 `사고 싶거나 팔고 싶어요`를 선택하고 사전 모드로 진입한다.
- 종목, 감정, 상황을 짧게 입력한 뒤 브릿지를 거쳐 AI 채팅으로 들어간다.
- AI는 공감 후 객관화 질문을 던지고, 마지막엔 `근거가 바뀌었는지 / 감정이 커졌는지`를 고르게 한다.
- 결과 화면에서는 비슷한 상황의 선택과 익명 한 줄 이유를 함께 보여준다.

**사용자가 얻는 것**
- FOMO 상태를 `지금 아니면 안 된다는 생각` 같은 일상 언어 태그로 인식한다.
- 감정과 근거를 구분한 뒤 `잠시 관망` 또는 `그래도 진행`을 스스로 선택한다.
- 선택 결과는 이후 패턴 기록에 활용된다.

---

### 시나리오 B. 하락장에서 급하게 손절하고 싶을 때

**상황**
- 사용자는 며칠째 하락 중인 보유 종목을 보며 손절 충동을 느낀다.
- 실제 근거 변화보다 `더 큰 손실이 올 것 같다`는 상상이 앞선다.

**사용자 목표**
- 지금 팔아야 하는지 정답을 듣는 것이 아니라,
- 현재 판단이 파국적 상상에 끌리고 있는지 확인하고 싶다.

**서비스 흐름**
- 사용자는 사전 모드로 진입해 종목, 감정, 상황을 입력한다.
- AI 채팅에서는 공감 후 `처음 세운 근거가 여전히 유효한지` 같은 질문으로 시선을 되돌린다.
- 메타 질문에서는 `근거 변화`와 `불안` 중 무엇이 더 큰 이유인지 고르게 한다.
- 결과 화면에서는 비슷한 상황의 관망 비율과 익명 이유를 보여준다.

**사용자가 얻는 것**
- 손절 충동이 단순 사실 반응이 아니라 `최악을 상상하는 패턴`일 수 있음을 본다.
- 감정이 큰 순간에 선택을 한 번 늦추고 다시 생각할 틈을 얻는다.

---

### 시나리오 C. 이미 한 매매가 마음에 걸릴 때

**상황**
- 사용자는 급등장에서 충동적으로 매수한 뒤, 금방 후회가 올라온 상태다.
- `왜 또 그랬지`라는 자책은 있지만, 그 순간 무엇이 판단을 흔들었는지는 정리되지 않았다.

**사용자 목표**
- 잘잘못을 평가받기보다,
- 방금 한 선택을 감정과 판단으로 나눠서 짧게 복기하고 기록으로 남기고 싶다.

**서비스 흐름**
- 사용자는 `이미 한 매매가 마음에 걸려요`를 선택하고 사후 복기 모드로 진입한다.
- AI는 공감 후 `근거가 먼저였는지 감정이 먼저였는지`를 돌아보게 하는 질문을 던진다.
- 결과 화면에 도달하면 이번 복기는 이미 저장된 상태로 보고,
- 사용자는 `멘탈 캘린더 보러 가기`를 통해 자신의 기록 흐름으로 이어진다.

**사용자가 얻는 것**
- 자책 대신 패턴 인식의 관점으로 방금 매매를 다시 보게 된다.
- 이번 복기가 다음 비슷한 순간에 떠올릴 수 있는 기록으로 남는다.

## 11. KPI

현재 단계의 KPI는 `정식 서비스 운영 지표`가 아니라, 해커톤 MVP가 핵심 가설을 검증했는지 보기 위한 기준으로 둔다.

### 11.1 핵심 가설

- 사용자는 투자 판단 직전/직후에 `감정과 근거를 분리해 보는 짧은 개입`을 필요로 한다.
- 사용자는 `투자 조언`보다 `질문 기반 판단 점검`을 받아들일 수 있다.
- 사용자는 결과 화면의 `집단 맥락 + 기록` 구조를 통해 선택을 마무리할 수 있다.

### 11.2 해커톤 검증 지표

| 지표 | 정의 | 확인 방식 |
| --- | --- | --- |
| **핵심 플로우 완주율** | 입력 시작 → 채팅 → 결과까지 한 번에 이어지는지 | 유저 테스트 / 내부 시연 |
| **사전/사후 분기 이해도** | 사용자가 두 모드 차이를 설명 없이 이해하는지 | 관찰 기반 테스트 |
| **결과 CTA 이해도** | 사전은 `선택 확정`, 사후는 `기록 보기`로 읽히는지 | 짧은 인터뷰 / 피드백 |
| **내 기록 연결 이해도** | 사용자가 내 기록이 멘탈 캘린더 중심 화면이라는 점을 이해하는지 | 시연 피드백 |

### 11.3 운영 전환 시 추후 볼 지표

정식 운영 단계에서만 아래 지표를 재정의해 사용한다.

- `쿨다운 전환율`
- `왜곡 유형별 PAUSE / PROCEED 분포`
- `사전 → 사후 회귀 흐름`
- `재방문율`

## 12. 개발 일정

| 시간 | 작업 내용 | 담당 |
| --- | --- | --- |
| **1단계** | PRD 정리, 핵심 유저 시나리오 확정, 입력/채팅/결과 플로우 정의 | 기획 / 전원 |
| **2단계** | 프론트 프로토타입 구현 (메인, 입력, 채팅, 결과, 내 기록) | FE / 디자이너 |
| **3단계** | 백엔드 계약 확정 (`question_type`, 결과 저장 시점, 집계용 데이터 구조) | BE / AI 엔지니어 |
| **4단계** | LLM 프롬프트 정리 및 API 연동 | AI 엔지니어 / BE |
| **5단계** | 더미/실데이터 혼합 상태로 유저 시나리오 점검, 버그 수정 | 전원 |
| **6단계** | 발표 자료 정리 및 시연 흐름 점검 | PM / 전원 |

**개발 우선순위 원칙: "데모에서 보여줄 것부터 만든다"**
1. 홈 → 입력(스텝 플로우) → AI 채팅(CBT 진단) → 결과(커뮤니티 데이터) 핵심 플로우 완성
2. 내 기록 (멘탈 캘린더)
3. 기록/집계 데이터가 화면 의미와 맞는지 확인

| 옵션 | 안정성 | 공수 | 역할 분리 |

```
  exploratory    → required: false → 건너뛰기 버튼 노출
건너뛰기 시:
  → [사용자가 이 질문을 건너뛰었습니다] 시스템 메시지 전달
  → AI 자연스럽게 다음 질문으로 진행
```
