# 🚀 TRADEMIND 기획안 v0.7

> **버전:** v0.7  
> **작성일:** 2026.04.02  
> **성격:** 현재 프로토타입 기준 최신 계획 문서  
> **이전 버전:** `docs/trademind-plan-v0.6.md`

---

## 1. 문서 목적

이 문서는 v0.6 이후 실제 프로토타입 작업에서 합의된 변경 사항을 반영한 최신 계획 문서입니다.

핵심 목적:

- 현재 화면 구조와 정보 구조를 최신 상태로 고정
- 결과 화면의 의미와 닫힘 방식을 명확히 정의
- 멘탈 캘린더 / 감정 공유 보드 / 선택 데이터의 연결 방식을 정리
- 이후 화면 수정 시 기준이 흔들리지 않게 함

---

## 2. 제품 진실

TRADEMIND는 범용 금융 앱이 아닙니다.

- 이 서비스는 `종목 추천`을 하지 않습니다.
- 이 서비스는 `매매 실행 최적화`를 목표로 하지 않습니다.
- 이 서비스는 투자자의 감정과 판단 사이를 분리해서 보게 하는 `투자 멘탈 케어 도구`입니다.
- 핵심 경험은 `사전 개입`과 `사후 복기`입니다.
- 핵심 차별점은 `CBT 기반 질문`과 `TOGETHER 데이터`, 그리고 `기록의 축적`입니다.

한 줄 정의:

> **TRADEMIND는 투자자의 감정적 판단을 늦추고, 스스로 이유를 다시 보게 돕는 AI 기반 멘탈 체크 서비스입니다.**

## 4. 현재 정보 구조

### 4.1 하단 탭 구조

현재 프로토타입 기준 하단 탭은 다음 철학으로 정리합니다.

- `홈`
- `대화하기`
- `인사이트`
- `설정`  
  - 설정은 현재 MVP 핵심이 아니며 placeholder 성격이 강함

중요:

- `대화하기` 라벨은 유지한다
- `입력`, `점검` 같은 시스템 용어보다 사용자 관점에서 더 자연스럽다
- UX writing 기준상 일상 언어와 서비스 톤에 더 적합하다

### 4.2 인사이트 구조

`멘탈 캘린더`와 `감정 공유 보드`는 별도 하단 메뉴 둘로 나누지 않는다.

이유:

- 둘 다 행동 전 핵심 플로우가 아니라 `돌아보기 / 집계 보기` 성격이다
- 둘을 묶는 상위 개념은 `인사이트`가 가장 자연스럽다
- 정보 구조상 하나의 탭 안에서 `내 기록`과 `함께 보기`를 넘나드는 쪽이 더 명확하다

현재 결론:

- 하단 탭 `인사이트` 하나
- 내부 탭 2개
  - `멘탈 캘린더`
  - `감정 공유 보드`

## 5. 화면 구조 원칙

### 5.1 입력 화면

입력 화면은 `롱폼`이 아니라 `스텝 플로우`다.

구조:

1. 0depth 진입 화면
2. 종목 단계
3. 감정 단계
4. 상황 입력 단계
5. 브릿지 단계

원칙:

- 0depth에서만 탭바 노출
- step 화면에서는 상단 step header + 하단 CTA
- 한 단계에 한 질문만 보여준다

### 5.2 AI 채팅 화면

사전/사후는 분기해야 하지만, 화면 파일을 둘로 나누지 않는다.

구조:

- 하나의 채팅 셸
- `pre / post` 시나리오 분기
- 대화는 누적 흐름처럼 보여야 함
- 프롬프트 응답 스키마를 프론트 view model로 변환해서 렌더링

### 5.3 결과 화면

결과 화면 역시 두 페이지로 찢지 않는다.

구조:

- 하나의 결과 셸
- `pre / post` variant 분기
- 요약 / TOGETHER / 하단 행동 구조는 공유
- 마지막 행동만 모드별로 다르게 가져감

### 5.4 인사이트 화면

인사이트는 하나의 목적지로 취급한다.

구조:

- 하나의 인사이트 셸
- 내부 탭으로 `멘탈 캘린더 / 감정 공유 보드`
- 읽기형 화면이므로 헤더는 고정하지 않는다

## 6. 최신 예상 화면 흐름

### 화면 1. 홈

- 현재 시장 감정 한 줄
- 핵심 CTA
- 익명 한 줄 미리보기
- 하단 탭 진입

### 화면 2. 입력

- `사고 싶거나 팔고 싶어요`
- `이미 한 매매가 마음에 걸려요`
- 종목 입력
- 감정 선택
- 상황 입력
- 브릿지

### 화면 3. AI 채팅

사전 모드:

- 공감
- 질문
- 메타 질문
- 선택

사후 모드:

- 공감
- 복기 질문
- 기록 마무리

### 화면 4. 결과

사전 모드:

- 현재 패턴 요약
- TOGETHER 데이터
- 익명 한 줄 이유
- `잠시 관망하겠습니다 / 그래도 진행하겠습니다`
- 결과 요약은 이미 준비된 상태로 보고
- 선택 직후 현재 화면 안에서 확인 상태로 전환

사후 모드:

- 이번 기록 요약
- TOGETHER 데이터
- 결과 화면 진입 시 이미 기록 저장 완료
- `멘탈 캘린더 보기` CTA
- 프로토타입에서는 현재 화면 안에서 짧은 전환 상태로 표현 가능

### 화면 5. 인사이트

내부 탭:

- `멘탈 캘린더`
- `감정 공유 보드`

멘탈 캘린더 해석 원칙:
- 히트맵은 감정 종류나 결과 평가를 보여주는 장치가 아니다
- `그날 기록이 남았는지`, 필요하면 `같은 날 기록이 여러 개였는지`를 날짜 흐름으로 보는 장치다
- 감정, 왜곡, 요약은 히트맵이 아니라 아래 리스트와 상세 카드에서 읽는다

## 7. 결과 화면 기획 보강

이 문서에서 가장 중요하게 다시 정의하는 영역입니다.

### 7.1 결과 화면의 역할

결과 화면은 단순 요약 화면이 아닙니다.

역할:

1. 사용자의 현재 판단을 다시 짧게 정리한다
2. 혼자가 아니라는 집단 맥락을 제공한다
3. 사용자의 선택을 하나의 행동으로 확정하게 만든다
4. 그 선택 또는 기록이 남는다는 감각을 준다

즉, 사전 결과의 CTA는 `선택 확정 버튼`, 사후 결과의 CTA는 `이미 저장된 기록으로 이동하는 버튼`입니다.

### 7.2 사전 결과의 두 CTA가 왜 필요한가

사전 결과의 CTA:

- `잠시 관망하겠습니다`
- `그래도 진행하겠습니다`

이 두 버튼은 추천/비추천이 아니라, 사용자의 현재 의사를 스스로 언어로 확정하게 만드는 장치입니다.

UX적 의미:

- 막연한 감정을 `선택 언어`로 바꾸게 함
- AI와 대화를 거친 뒤에도 사용자가 주체라는 점을 유지
- 결과를 기록 가능한 행동 데이터로 변환

### 7.3 왜 둘 다 같은 방식으로 닫혀야 하는가

현재 합의:

- `관망`과 `진행`은 서로 다른 판단이다
- 하지만 UX 구조는 동일하게 닫는다

이유:

- 한쪽만 특별 취급하면 서비스가 정답을 암시하는 것처럼 보인다
- 투자 조언 서비스처럼 읽히는 위험을 피해야 한다
- 같은 인터랙션 구조를 유지해야 사용자가 평가받는 느낌이 줄어든다

### 7.4 선택 후 흐름

사전 결과:

- 버튼 클릭
- 선택값 저장
- 현재 화면 안에서 확인 상태로 전환
- 별도 화면 추가 없음

사후 결과:

- 결과 화면 진입 시 기록 저장 완료
- 버튼 클릭
- 멘탈 캘린더로 이어지는 이동 액션
- 프로토타입에서는 현재 화면 안에서 짧은 전환 상태로 표현 가능

즉, 공통 원칙은 다음과 같습니다.

> **사전 모드는 버튼을 누른 직후 현재 화면 안에서 선택 확인 상태로 닫고, 사후 모드는 이미 저장된 기록을 다음 화면으로 이어주는 CTA로 읽히게 만든다.**

### 7.5 진행 버튼의 제품 태도

`그래도 진행하겠습니다`는 반드시 필요합니다.

없애면 안 되는 이유:

- 제품이 사용자의 판단을 대신 결정하려는 느낌이 강해짐
- 서비스가 너무 교정적이고 통제적으로 보일 수 있음
- 현실적으로 사용자는 항상 관망만 하지 않음

하지만 이 버튼은 이렇게 다뤄야 합니다.

- 막지 않는다
- 응원하지도 않는다
- 기록은 남긴다
- 필요하면 나중에 사후 복기로 돌아올 수 있다는 암시만 남긴다

한 줄로 정리하면:

> **진행은 허용하되, 지지하지 않고, 기록과 복기의 가능성만 남긴다.**

### 7.6 확장 기능으로 남겨둘 것

`잠시 관망하겠습니다` 이후의 이유 점검 화면은 아이디어 자체는 좋다.

하지만 현재 판단:

- MVP에서는 제외
- 결과 직후 별도 화면 2장으로 늘리는 것은 과함
- 확장 시에도 `1-step 후속 질문` 정도로만 검토

## 8. 결과 데이터와 대시보드 활용 가능성

결론부터 말하면, 활용 가능성은 높습니다.

단, 이것은 `성과 대시보드`가 아니라 `행동 패턴 대시보드`여야 합니다.

### 8.1 결과 데이터가 의미 있는 이유

결과 CTA는 단순 UI 상태가 아니라, 아래를 보여주는 행동 데이터입니다.

- 사전 개입 후 실제로 멈출 의사가 있었는지
- 어떤 왜곡 유형에서 `진행` 선택이 많이 나오는지
- 사후 복기로 되돌아오는 패턴이 어떤지

이건 이후 인사이트 화면에서 매우 유용합니다.

### 8.2 대시보드에서 가능한 지표

#### 개인 지표

- `쿨다운 전환율`
  - 사전 모드 진입 대비 `PAUSE` 비율
- `진행 선택 비율`
  - 어떤 상황에서 계속 밀고 가는지
- `왜곡 유형별 선택 패턴`
  - 예: `지금 아니면 안 된다는 생각`에서 `PROCEED` 비율이 높은지
- `사전 -> 사후 회귀 흐름`
  - `PROCEED`를 선택한 뒤 나중에 `POST` 복기로 다시 들어오는지
- `시간대/요일 패턴`
  - 언제 감정적 선택이 몰리는지

#### 집단 지표

- 같은 왜곡 유형에서 `PAUSE / PROCEED` 비율
- 특정 감정 상태의 빈도
- 사후 후회 비율

### 8.3 보여줘도 되는 것과 안 되는 것

보여줘도 되는 것:

- `반복되는 패턴`
- `선택 분포`
- `시간대 경향`
- `같은 상황의 다른 사람들 선택`

보여주면 안 되는 것:

- `관망이 더 좋은 선택이었다`
- `진행이 잘못된 선택이었다`
- 수익률 기반의 우월감/열등감
- 사용자끼리의 순위 비교

원칙:

> **대시보드는 사용자의 투자 성과를 평가하는 도구가 아니라, 반복되는 감정과 행동 패턴을 비추는 거울이어야 한다.**

## 9. 데이터 저장 원칙

UI 문구와 내부 저장값은 분리한다.

현재 방향:

- `PAUSE`
- `PROCEED`
- `POST_RECORDED`

이렇게 내부 enum으로 저장하고, 실제 버튼 문구는 UX 라이팅 기준에 맞춰 바꿀 수 있게 둔다.

이게 필요한 이유:

- 카피 수정이 자유로워짐
- 집계가 쉬워짐
- 대시보드 설계가 안정적이 됨

## 10. 현재 프로토타입 기준 반영 사항

v0.7은 아래 현재 구조를 최신 합의안으로 본다.

- 입력 화면은 스텝 플로우다
- 입력 0depth에서만 탭바가 보인다
- AI 채팅은 하나의 셸 안에서 pre/post 시나리오가 분기된다
- 결과 화면은 하나의 셸 안에서 pre/post variant가 분기된다
- 결과 CTA는 선택 후 현재 화면 안에서 확인 상태로 닫힌다
- 인사이트는 하나의 화면 안에서 멘탈 캘린더 / 감정 공유 보드 탭으로 묶인다
- 읽기형 화면인 결과 / 인사이트에서는 헤더를 고정하지 않는다

## 11. 아직 열어둘 질문

### 11.1 `그래도 진행하겠습니다`의 정확한 의미

현재는 `실행 의사`로 해석한다.

다만 추후 다음 둘 중 어느 쪽으로 더 정확히 정의할지는 검토 여지가 있다.

- 실제 매수/매도 실행 의사
- 아직 그 방향으로 마음이 기운 상태

현재 MVP에서는 전자를 기준으로 보되, 투자 실행 완료 여부까지는 추적하지 않는다.

### 11.2 결과 확인 상태 이후 최종 CTA

현재는 구조만 확정했고, 목적지 CTA는 아직 가볍게 열어둔다.

후보:

- `메인으로 돌아가기`
- `대화 마치기`
- `멘탈 캘린더 보기`

원칙은 하나다.

- `관망`과 `진행`은 동일한 구조로 닫힌다

### 11.3 설정 탭

현재 정보 구조상 `settings`는 우선순위가 낮다.

향후 판단:

- MVP에서 placeholder 유지
- 또는 제거 후 3탭 구조로 단순화

## 12. v0.6 → v0.7 핵심 변경 요약

1. 결과 화면을 `사전/사후 별도 페이지`가 아니라 `하나의 셸 + variant`로 정리
2. 결과 CTA를 `선택 확정 + 기록 확인` 단계로 재정의
3. `관망 / 진행`은 서로 다른 판단이지만 `같은 방식으로 닫는다`는 원칙 확정
4. 인사이트 화면을 `멘탈 캘린더 + 감정 공유 보드` 통합 구조로 정리
5. 하단 탭의 `대화하기` 라벨 유지 결정
6. 결과 선택 데이터의 대시보드 활용 방향을 `행동 패턴 인사이트`로 명시
