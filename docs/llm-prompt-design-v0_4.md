# TRADEMIND LLM Prompt Design Guide

> 버전: v0.4  
> 작성일: 2026.04.03  
> 기반 문서: `docs/llm-prompt-design.md`, `docs/TRADEMIND_PRD_v1.3.md`

---

## 1. 문서 목적

이 문서는 현재 사용 중인 LLM 계약을 뒤엎지 않고,
PRD v1.2와 충돌하지 않도록 **운영 해석을 보완**하는 문서다.

즉:
- 현재 JSON 구조는 유지
- `question_type` 계약 유지
- `required` 필드도 현재 구현과 연동된다면 유지 가능
- 다만 어떤 값이 핵심 계약이고, 어떤 값은 보조 해석인지 분명히 한다

---

## 2. 지금 유지하는 계약

현재 유지하는 핵심 계약:
- 입력: `mode`, `ticker`, `emotion`, `text`
- 출력:
  - `distortion_type`
  - `distortion_tag`
  - `empathy`
  - `question`
  - `question_type`
  - `meta_question`
  - `meta_question_type`
  - `meta_options`

현재 구현과의 연결을 위해 `required`를 문서에 남겨둘 수 있다.  
다만 이것은 **질문의 본질적 의미를 담는 필드가 아니라, 백엔드/프론트 처리용 파생값**으로 본다.

---

## 3. 해석 우선순위

우선순위는 아래처럼 둔다.

1. `question_type`
2. `meta_question_type`
3. `mode + stage`
4. `required`

의미:
- 질문의 성격은 `question_type`이 정의한다
- 화면 흐름은 `mode + stage`가 정의한다
- `required`는 최종 UI 제어를 위한 파생값이다

즉 `required`가 문서에 남아 있어도,
이 계약의 중심축은 여전히 `question_type`이다.

### 3.1 stage 정의

`stage`는 프론트가 채팅 흐름을 제어하기 위해 내부적으로 관리하는 단계값이다.
LLM이 반환하는 값이 아니라, 프론트가 AI 응답을 받을 때마다 증가시키는 카운터다.

| mode | stage | 의미 |
| --- | --- | --- |
| `pre` | `1` | 첫 번째 AI 응답 (공감 + 탐색 질문) |
| `pre` | `2` | 두 번째 AI 응답 (메타인지 질문 + 선택지) |
| `post` | `1` | 첫 번째 AI 응답 (공감 + 복기 질문) |

흐름 제어 원칙:
- `mode = pre, stage = 1`: `question_type = exploratory` → 건너뛰기 가능
- `mode = pre, stage = 2`: `meta_question_type = meta_cognition` → 버튼 선택 필수, 결과 화면으로 이동
- `mode = post, stage = 1`: `question_type = exploratory` → 건너뛰기 가능, 이후 결과 화면으로 이동

즉 `stage`는 "지금 몇 번째 AI 응답인가"를 추적하는 값이며,
프론트는 이 값을 기준으로 다음에 결과 화면으로 넘어갈지 판단한다.

## 4. 현재 구현 기준 정리

### 4.1 입력 스키마

현재 입력은 단일 감정 선택 기준으로 읽는다.

```json
{
  "mode": "pre",
  "ticker": "삼성전자",
  "emotion": "불안해요",
  "text": "3일 연속 빠지는데 더 떨어질 것 같아요..."
}
```

주의:
- `emotion`은 현재 구현 기준 단일값이다
- `emotions[]` 복수 배열로 되돌려 읽지 않는다

### 4.2 질문 타입

| 필드 | 값 | 의미 |
| --- | --- | --- |
| `question_type` | `exploratory` | 첫 질문. 탐색형 질문 |
| `meta_question_type` | `meta_cognition` | 메타인지 질문. 최종 선택 전 단계 |

### 4.3 required 해석

`required`는 현재 문서에 남아 있더라도, 아래처럼 읽는다.

- `exploratory`
  - 보통 건너뛰기 가능
- `meta_cognition`
  - 보통 선택 필요

즉 `required`는 직접 설계의 중심값이 아니라,
`question_type`을 UI 처리로 풀어낸 결과값이다.

## 5. PRD v1.2와 맞춘 주의사항

### 5.1 결과 데이터 의미와 직접 연결하지 않는다

LLM 계약은 결과 화면의 최종 저장값을 직접 정의하지 않는다.

즉 아래는 LLM 책임이 아니다.
- `PAUSE`
- `PROCEED`
- `POST_RECORDED`

이 값들은 결과 화면과 기록 로직의 책임이다.

LLM은 어디까지나:
- 왜곡을 분류하고
- 질문을 생성하고
- 메타인지 선택지를 제시하는 역할까지만 맡는다

### 5.2 멘탈 캘린더 표시값을 LLM 출력에서 직접 가져오지 않는다

멘탈 캘린더 히트맵의 메인 표시값은
- 그날 기록이 남았는지
- 같은 날 기록이 여러 개였는지

즉 히트맵은 상세 내용을 설명하는 장치가 아니라,
기록이 남은 날짜 흐름을 한눈에 보는 장치다.

따라서 LLM 문서에서 결과값을 해석할 때도
- `관망 / 진행 / 기록`을 동일한 축으로 읽도록 유도하면 안 되고
- 감정/왜곡/요약을 히트맵 셀 값으로 직접 매핑하도록 유도해도 안 된다

감정, 왜곡, 요약은
- 날짜 상세 카드
- 최근 기록 리스트

같은 영역에서 읽는 정보로 내려가는 것이 적절하다.

### 5.3 사전/사후의 책임을 구분한다

사전 모드
- 사용자가 최종 선택을 내리기 전까지 질문을 통해 생각을 분리하게 만드는 흐름

사후 모드
- 이미 한 행동을 짧게 복기하고 기록으로 이어지게 만드는 흐름

즉 사후 모드에서는
LLM이 마지막 행동 결정을 다시 유도하는 구조로 길게 끌지 않는다.

## 6. 문서 유지 원칙

현재 프로젝트는 이미 진행 중이므로,
이 문서는 기존 LLM 계약을 대체하지 않는다.

이번 버전의 목적은 아래에 한정한다.
- 기존 계약 유지
- PRD v1.2와의 충돌 해소
- 구현 중인 프론트/백엔드 해석 기준 보완

특히 멘탈 캘린더 관련해서는
- LLM 출력이 히트맵 값을 직접 결정하지 않고
- 기록 존재 여부/기록 수는 프론트 또는 백엔드에서 계산하며
- LLM 출력은 상세 해석용 데이터로만 사용된다는 점을 유지한다.

추후 실제 계약을 줄이거나 바꿀 필요가 생기면,
그때 별도 버전에서 다시 정리한다.

## 7. 이번 버전에서 추가된 내용 (v0.3)

- `required`를 중심 계약이 아닌 파생값으로 재정의
- `emotion` 단일값 해석 재고정
- LLM 계약과 결과 저장값의 책임 분리 명시
- PRD v1.2의 멘탈 캘린더 해석과 충돌하지 않도록 운영 주의사항 추가
