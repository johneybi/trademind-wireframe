## 3. 해석 주의사항

### 3.1 `PAUSE / PROCEED / POST_RECORDED`는 같은 축이 아니다

이 셋은 같은 레벨의 사용자 상태값으로 보면 안 된다.

- `PAUSE`
  - 사전 모드에서의 최종 선택
- `PROCEED`
  - 사전 모드에서의 최종 선택
- `POST_RECORDED`
  - 사후 모드에서 복기 기록이 저장된 완료 상태

즉:
- `PAUSE / PROCEED`는 사전 선택 데이터
- `POST_RECORDED`는 사후 완료 상태

따라서 이 셋을 한 차트, 한 캘린더, 한 분류축으로 직접 비교하는 것은 부적절하다.

### 3.2 멘탈 캘린더의 메인 표시값으로 쓰지 않는다

`PAUSE / PROCEED / POST_RECORDED`는 멘탈 캘린더의 메인 표시값으로 쓰지 않는다.

멘탈 캘린더 히트맵은 우선적으로 아래를 보여줘야 한다.
- 기록이 남은 날짜의 흐름
- 어느 시기에 점검/복기가 몰렸는지
- 필요하면 같은 날 복수 기록이 있었는지를 진하기 차이로만 표현하는 것

반대로 `PAUSE / PROCEED / POST_RECORDED`는
- 날짜 상세
- 세션 상세
- 집계용 보조 메타데이터

정도로만 쓰는 것이 적절하다.

또한 감정, 인지 왜곡, 요약 텍스트는
- 히트맵 셀 자체의 값이 아니라
- 날짜 상세 카드나 최근 기록 리스트에서 읽는 정보로 두는 것이 적절하다.

### 3.3 내부적으로는 이렇게 읽는다

현재 스키마를 바꾸지 않는 전제에서, 내부 해석은 아래처럼 분리해서 이해한다.

- `session_mode = PRE | POST`
- `user_decision = PAUSE | PROCEED | POST_RECORDED`

하지만 의미상으로는 다음과 같이 읽는다.

- `PRE + PAUSE`
  - 사전 점검 후 관망 선택
- `PRE + PROCEED`
  - 사전 점검 후 진행 선택
- `POST + POST_RECORDED`
  - 사후 복기 저장 완료

즉 `user_decision` 단독으로 분석하지 않고, 반드시 `session_mode`와 함께 읽는다.

## 4. 결과 저장 시점 해석

### 4.1 사전 모드

사전 모드에서는 결과 화면 진입 시점과 최종 선택 시점을 분리해서 읽는다.

- 결과 화면 진입
  - 분석 결과가 준비된 상태
- 최종 버튼 클릭
  - `PAUSE` 또는 `PROCEED` 확정

즉 `PAUSE / PROCEED`는 결과 화면 진입 시점이 아니라 **최종 선택 시점**의 데이터다.

### 4.2 사후 모드

사후 모드에서는 결과 화면 진입 시점에 이미 복기 저장이 완료된 상태로 읽는다.

- 결과 화면 진입
  - `POST_RECORDED` 상태로 간주
- `멘탈 캘린더 보러 가기`
  - 저장이 아니라 이동 CTA

즉 사후 결과 CTA는 저장 버튼이 아니다.

---

## 5. 대시보드 / 집계 원칙

의미 있게 쓸 수 있는 지표:
- 사전 모드 진입 대비 `PAUSE` 비율
- 사전 모드 진입 대비 `PROCEED` 비율
- 왜곡 유형별 `PAUSE / PROCEED` 분포
- `PROCEED` 후 나중에 `POST` 세션으로 돌아오는 비율

의미 없거나 오해를 만드는 지표:
- `PAUSE / PROCEED / POST_RECORDED`를 한 축으로 놓고 비율 비교
- 멘탈 캘린더에서 세 값을 메인 상태로 색칠

---

## 6. 백엔드 전달용 주의사항

이 문서의 목적은 스키마 변경 요청이 아니다.

백엔드 구현에서는 아래만 주의하면 된다.

- `user_decision`은 단독 해석하지 말고 `session_mode`와 함께 해석
- 사후 모드 CTA는 저장 버튼이 아니라 이동 버튼으로 이해
- 멘탈 캘린더 히트맵은 일별 기록 존재 여부 또는 기록 수 기반으로 계산
- 감정/왜곡/요약은 상세 응답 데이터로 내려주고, `user_decision`은 보조 메타로만 사용
